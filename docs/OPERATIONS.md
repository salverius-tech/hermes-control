# Operations Runbook

## Local backend

```bash
python -m venv .venv
source .venv/Scripts/activate
python -m pip install -r requirements.txt
CONTROL_API_TOKEN=dev-token CONTROL_API_DB_PATH=./data/control-api.db uvicorn services.control_api.main:app --host 0.0.0.0 --port 8787
```

Use a real secret outside source control for anything beyond local development.

To execute submitted tasks with a local Hermes command, add `CONTROL_API_HERMES_COMMAND`. The task prompt is sent on stdin.

```bash
CONTROL_API_HERMES_COMMAND='hermes chat -q'
```

To send Discord webhook notifications for task approvals and terminal task states, add `CONTROL_API_DISCORD_WEBHOOK_URL` with a webhook URL stored outside source control.

```bash
CONTROL_API_DISCORD_WEBHOOK_URL='https://discord.com/api/webhooks/[REDACTED]'
```

Notifications are best-effort and never block task execution. The diagnostics endpoint reports `notification_mode` as `discord` when this value is configured and `disabled` otherwise.

## Mobile development

```bash
cd apps/mobile
npm install
npm run typecheck
npm start
```

For physical Android devices, configure the mobile app with the host LAN IP/Tailscale hostname, not `127.0.0.1`.

## Verification commands

Fast backend-only check:

```bash
source .venv/Scripts/activate
python -m pytest services/control_api/tests -v
```

Full non-device canonical check:

```bash
source .venv/Scripts/activate
python scripts/verify.py
```

Android release build:

```bash
python scripts/verify.py --android
```

GitHub Actions release APK artifact:

1. Open the `verify` workflow in GitHub Actions.
2. Choose **Run workflow**.
3. Set `build_android=true`.
4. Download the `hermes-mobile-control-release-apk` artifact after the run completes.

Connected-device release install/launch check:

```bash
python scripts/verify.py --android --sideload
```

Connected-device release install/launch plus Maestro UI smoke flow:

```bash
python scripts/verify.py --maestro
```

The Maestro flow lives at `.maestro/smoke.yaml`. It verifies dashboard text, bottom navigation, Settings gear navigation, and the absence of the old redundant dashboard action buttons.

If sideload fails with `no devices/emulators found`, unlock the phone, connect USB, enable USB debugging, and rerun `adb devices` until the device appears as `device`.

## Release/debug APK notes

- Use release APKs for phone review because the JS bundle is embedded.
- Debug APKs may require Metro and can show splash/dev-client behavior if Metro is unavailable.
- Release builds need the Gradle memory settings in `apps/mobile/android/gradle.properties`.

## Troubleshooting

### API returns `401`

Check that the backend `CONTROL_API_TOKEN` matches the token saved in the mobile Settings screen.

### Mobile cannot connect from phone

Use the host machine LAN IP/Tailscale hostname and ensure the backend binds to `0.0.0.0`.

### Voice prompt button asks for permission or does nothing

Speech recognition uses `expo-speech-recognition`; native configuration is generated by Expo prebuild. Re-run native build verification after dependency/plugin changes.

### Android release build fails with memory/dexing errors

Stop Gradle and retry with the committed JVM settings:

```bash
cd apps/mobile/android
./gradlew --stop
./gradlew assembleRelease
```

### Tasks complete with an unconfigured-adapter message

Set `CONTROL_API_HERMES_COMMAND` to the local Hermes command you want the companion API to run. Without it, the API intentionally records a completed dry-run lifecycle so mobile clients can test task timelines without invoking Hermes.
